# Phase 2 v1 surfaces trusted previews in a cross-origin sandboxed iframe; the untrusted pixels/neko lane is deferred

The source-plan security model (§4) gives two surfacing **trust lanes**: a **trusted iframe** for *"your own trusted preview servers"* and a **pixels lane** (neko/WebRTC video texture, or screenshots) for *"arbitrary running apps."* Phase 2 v1 ships **only the trusted-iframe lane**. Every Preview spec carries a `trust` level (`trusted | untrusted`) so the taxonomy lives in the model from day one, but **only `trusted` specs are bootable in v1**; `untrusted` specs are accepted into the registry and shown as *"pixels lane not yet available."* The neko/WebRTC (or headless-screenshot) pixels lane — a heavy streaming subsystem — is **deferred**.

The hard rule still holds: *the cockpit never executes agent-authored code in its own origin.* A trusted preview is rendered at the container's own origin (`http://127.0.0.1:<allocated-port>` — a distinct origin from the cockpit, so the Same-Origin Policy is the **primary** defense, not an error boundary), in a `sandbox`-attributed iframe, with **zero Hub credentials** ever crossing the boundary. "Trusted" therefore only ever means *"we will render its DOM in an isolated sandbox,"* never *"we grant it ambient access."* Because the frame is cross-origin to the embedder, `allow-scripts` + `allow-same-origin` is safe (the sandbox-escape only applies to frames same-origin with their parent).

We chose this scoping because §4 itself frames the iframe-for-trusted-servers path as the v1 deliverable (*"get a real iframe **in v1**"*) and the arbitrary-app pixels path as the harder later story; shipping only the iframe lane keeps Phase 2 chunkable and CI-testable like Phase 0/1 while still delivering real isolated, disposable, container-per-boot previews of your own output. Consequences: there is **no postMessage app-protocol in v1** (the trusted iframe is display-only; the channel lands with the pixels lane), and framing `http://127.0.0.1:<port>` inside the **packaged Tauri** webview (custom scheme + CSP/mixed-content) is a documented verify-later caveat, exactly as Phase 1 flagged webview mic-permission.
