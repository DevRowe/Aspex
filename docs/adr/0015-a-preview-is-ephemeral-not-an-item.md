# A Preview is an ephemeral, disposable concept — not an Item; the world-model stays untouched

An **Item** is defined (CONTEXT.md, ADR-0001) as a *persistent* thing that may need attention — exactly one per real-world object, upserted in place, never a point-in-time record. A **Preview** is the opposite: ephemeral, disposable, *"the persistent monitoring layer stays untouched; previews are disposable"* (source plan §5). So a Preview must **not** be an Item. It is a new first-class concept — never persisted to the world-model, never ranked, never in needs-me. Booting one is **always an explicit user action**; there is no auto-boot.

It is also not an Action (Actions are risk-tiered operations dispatched through an adapter to an upstream Source — booting a local container is neither) and is richer than a Deep-link (a passive URL). We therefore add a separate `Preview` / `PreviewSpec` type family in `packages/schema` (the way `voice.ts` was added in Phase 1), and **leave `AttentionItem`/`types.ts` completely unchanged**. Previewability is surfaced by the **client cross-referencing** the spec registry's optional `itemId` binding against the Items it renders — an Item card shows a "Preview" affordance when a spec binds to its id. The Hub never annotates Items with preview state.

We considered adding an optional `preview?` field onto `AttentionItem` (sibling to `deepLink`), but rejected it: the binding lives in the registry and no adapter sets it, so the field would be empty on every adapter-emitted Item and back-filled by an overlay — dirtying the core type that "stays untouched." Keeping `AttentionItem` pure and cross-referencing on the client is strictly more faithful to the disposable-previews invariant.
