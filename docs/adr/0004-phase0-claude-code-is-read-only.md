# Phase 0 claude-code is read-only; github is two-way; PTY injection is deferred

The Phase 0 Definition of Done says "send a response/approval through the right adapter," but for claude-code that means injecting input into a blocked interactive session via PTY — which §8 of the plan marks experimental, grey-area, and dependent on subscription interactive billing that "changes... don't architect around it." We will not put that mechanism inside the releasable "boring but excellent" core. **Phase 0: the claude-code adapter is read-only — it surfaces blocked/errored sessions and offers a deep-link / "focus this terminal" affordance, and you answer in your own terminal. The github adapter is fully two-way (approve / merge / comment / re-run) because those are safe REST calls.** PTY injection moves to a later, explicitly-experimental step.

We rejected building PTY control into Phase 0 (honors the DoD literally but makes the stable core depend on a fragile, grey-area path a weaker implementer can't build robustly) and rejected a fully read-only Phase 0 (simplest, but strips the "approve safely" value prop from the first release). Consequence: the Phase 0 DoD is reworded — "respond" is two-way for github and observe-and-jump for claude-code.
