# The Preview Deck boots declared preview specs; it never builds, checks out, or computes what to run

Aspex is **not an orchestrator** — it never runs agents, manages worktrees, computes diffs, schedules, or runs CI; it *consumes* upstream tools. The Phase 2 Preview Deck "boots container-isolated ephemeral previews," which looks like orchestration. We draw a hard line: the Preview Deck **boots a declared Preview spec** — an already-built image or compose file the user (or, later, an adapter) has declared — and does only *boot → isolate → show → dispose*. It **never** builds the app, resolves a branch, checks out a worktree, or computes what to run. Pulling a declared image is allowed; building is not.

In v1 the specs come from a **local registry** the user maintains in `~/.aspex/` config (outside the repo, like the token and DB), each spec optionally bound to an Item by id. **Adapter-surfaced specs** (e.g. a future claude-code adapter that knows a session's workdir has a compose file) are a deliberately stubbed extension point — the [[Provision]] pattern, wired later — so the whole Preview Deck v1 sits *beside* the Phase 0 adapters and touches none of them.

We chose this boundary because "container-isolated preview" is the single Phase 2 feature that most threatens Aspex's identity: if it built and ran projects it would re-implement container-use / vibe-kanban and fail the one-sentence test (*does this improve how I perceive and direct agents?*). Booting a *declared, pre-built* spec keeps "container per boot, disposable" (source plan §4) while staying firmly on the consume-not-orchestrate side. Consequence: a spec whose "boot" is merely "attach to this already-running URL" is just the degenerate case of the same model.
