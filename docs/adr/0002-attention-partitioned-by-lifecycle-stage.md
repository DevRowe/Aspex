# Attention is partitioned by lifecycle stage across adapters

The github adapter is the "universal" surface for any agent that opens a PR, so it overlaps with the per-agent adapters (claude-code, codex) that observe the same work. To stop one unit of work from double-surfacing in needs-me without fragile session↔PR correlation, we partition which adapter owns attention by lifecycle stage: **per-agent adapters raise `attentionRequired` only for in-flight states GitHub cannot see — `blocked` (waiting on a human) and `error`; the github adapter owns the PR lifecycle — review-requested, failing CI, and awaiting-merge.** Session completion (`done`/`Stop`) is Ambient (informational), not a needs-me card; the attention has moved to the PR.

We chose this over soft-correlating sessions to PRs by branch name (more accurate but error-prone for a less-capable implementer) and over showing both cards (simplest but violates calm-by-default). The known cost: a Claude Code session that finishes without opening a PR produces only an Ambient Item, so finished *local-only* work does not demand attention — it must be noticed in the ambient list. If that proves wrong in practice, the fix is to add soft correlation later, which does not change the Item model.
