# Liveness is two-track: poll-health for polled sources, heartbeats for push sources

Liveness ("how much we trust an Item's reported state") is computed differently per adapter shape, because polled and push sources fail in opposite ways. **Polled sources (github):** liveness reflects the Hub's own poll health — Items stay `live` while the last poll cycle succeeded, and the whole source degrades `quiet`→`stale`→`lost` only when polling fails or is rate-limited. A freshly-polled PR that has sat idle for days is still `live`, because our picture of it is current. **Push sources (claude-code):** liveness reflects heartbeat freshness; the adapter synthesizes heartbeats from a high-frequency hook (`PostToolUse`) so a heads-down working agent stays `live` instead of rotting to `stale`. **Terminal states** (`done`/`error` after `Stop`) are confirmed-final and never decay.

We rejected a single wall-clock decay (one rule, but it mislabels idle-but-fresh polled Items as stale and quietly-working agents as lost — the exact silent-wrongness the product must avoid) and rejected no-synthetic-heartbeats (honest but alarmingly noisy for normal long runs). The cost is that the claude-code adapter must install `PostToolUse` purely as a heartbeat, which the chunked build plan has to call out explicitly.
